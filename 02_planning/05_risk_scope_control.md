# 리스크와 범위 통제

## 주요 리스크

| 리스크 | 설명 | 대응 |
|---|---|---|
| 20개 모듈 전부 실제 작동 | 개발량 과다 | MVP 6개 + 나머지 설계 문서/더미 UI |
| 외부 사이트 의존 | VS Code 에이전트 인터넷 불가 | 자체 구현 또는 링크 검증 문서 제공 |
| 딥러닝 모델 무거움 | 브라우저 성능 저하 | 사전 학습 경량 모델, 샘플 기반 대체 |
| 초등 개인정보 | 카메라·음성 데이터 | 기본은 샘플 데이터, 선택 시 명시적 동의 안내 |
| 개념 단순화 오류 | 쉬운 설명이 틀린 설명이 될 수 있음 | 용어집과 오개념 방지 섹션 포함 |
| 생성형 AI 과신 | 성인·학생 모두 환각 위험 | 결과 검증 활동 필수 포함 |

## 범위 통제 원칙

- 모든 모듈을 완성하려 하지 않는다.
- 먼저 “보이는 작동”을 만든다.
- 모델 성능보다 학습 가능한 설명력을 우선한다.
- 고급 기능은 실습 링크가 아니라 자체 축소 모형으로 대체한다.
- 각 모듈은 1차시 수업에서 쓸 수 있는 단위로 제한한다.

## 구현 난도 분류

| 난도 | 예시 | 전략 |
|---|---|---|
| 낮음 | BFS/DFS, K-Means | 직접 구현 |
| 중간 | 결정경계, 토큰화, attention mock | 직접 구현 + 시각화 |
| 높음 | MNIST CNN, RNN/LSTM, RL | 경량 모델 또는 축소 구현 |
| 연구형 | 에이전틱 AI, 피지컬 AI, VLA | 개념 시뮬레이션과 외부 연계 안내 |
